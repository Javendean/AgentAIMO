# Known Unknowns

This document is a critical list of important questions that the current code/docs do NOT answer with confidence.

- Blindspot ID: BS-UNK-001
- Title: Does Kaggle Enforce Offline Wheels?
- Category: Known Unknowns
- Severity: high
- Confidence: low
- Status: unresolved unknown
- Why this is a blindspot: The deployment script `notebook/kaggle_notebook.py` uninstalls packages and reinstalls vLLM from `aimo-vllm-wheels`. We do not know if this is strictly required for the current Kaggle environment or if it is legacy logic.
- Evidence: `notebook/kaggle_notebook.py` Cell 1.
- What could be misleading: A developer assumes they need to recreate this exact environment locally, when in fact it might be a workaround for an older Kaggle container image.
- Potential consequence: Wasted time maintaining offline wheels and debugging environment setups.
- How to verify or falsify it: Remove the offline wheel installation logic in a private Kaggle notebook and attempt to run the pipeline with the default environment.
- Recommended next step: Attempt to deploy the code without the offline wheels and document the required environment dependencies clearly.
- Blocks future work? no

- Blindspot ID: BS-UNK-002
- Title: Is the `.jsonl` data tainted?
- Category: Known Unknowns
- Severity: high
- Confidence: low
- Status: unresolved unknown
- Why this is a blindspot: Multiple `.jsonl` files exist in the root directory (e.g., `research_data.jsonl`, `research_data2.jsonl`, `dpo_correction_dataset_v4.jsonl`). It is entirely unclear whether these are historical backups, actively consumed inputs, or outputs from the current `analyze_results.py` pipeline.
- Evidence: `docs/LEGACY_ARTIFACT_INVENTORY.md` lists these artifacts.
- What could be misleading: A developer or automated agent might assume `research_data.jsonl` is the canonical output of a recent sprint and base their optimizations on it, when in reality it might be corrupted or generated by a divergent version of the agent.
- Potential consequence: Optimization cycles and DPO training might be based on noisy, outdated, or fundamentally flawed artifact data, polluting the results.
- How to verify or falsify it: Trace the file `dpo_correction_dataset.jsonl` through all active pipeline scripts to determine if it is generated or consumed.
- Recommended next step: Move generated datasets to a dedicated `data/processed/` or `data/raw/` directory and update scripts to use parameterized paths rather than hardcoded root paths.
- Blocks future work? no

- Blindspot ID: BS-UNK-003
- Title: Does GenSelect Handle Truncation Well?
- Category: Known Unknowns
- Severity: high
- Confidence: low
- Status: unresolved unknown
- Why this is a blindspot: When a weak consensus is reached, the agent truncates reasoning traces to 2000 characters before asking an LLM to select the "best" one.
- Evidence: `agent/deep_researcher.py:_genselect`: `truncated_sol = attempt.solution_text[-2000:] if len(attempt.solution_text) > 2000 else attempt.solution_text`
- What could be misleading: By arbitrarily truncating the solution, the GenSelect model might be evaluating a complex, 4000-token proof based solely on its final 2000 characters, losing the crucial setup or premises.
- Potential consequence: The LLM judge is forced to evaluate incomplete information, leading to blind or hallucinated choices.
- How to verify or falsify it: Inspect the GenSelect prompt generated for a very long trace and observe whether critical context is missing.
- Recommended next step: Summarize the trace intelligently using an LLM or preserve the beginning and end of the trace rather than an arbitrary 2000-character tail.
- Blocks future work? no

- Blindspot ID: BS-UNK-004
- Title: Is the Fail-Fast Swap to 72B tested?
- Category: Known Unknowns
- Severity: high
- Confidence: low
- Status: unresolved unknown
- Why this is a blindspot: `notebook/kaggle_notebook.py` includes logic to delete the 120B model and swap to a 72B model if the canary test fails. We do not know if this complex state-transition has ever been robustly tested.
- Evidence: `notebook/kaggle_notebook.py:run_canary_test` and the subsequent "KILL SWITCH" and "SWAP TO QWEN-72B" logic.
- What could be misleading: A developer running the package code locally might assume they are running the identical fault-tolerant system used on Kaggle, but they are missing the canary test entirely.
- Potential consequence: The package code is brittle to model logic collapse or hardware OOMs, while the notebook deployment script attempts to handle them. This creates a local-vs-production mismatch.
- How to verify or falsify it: Search for `run_canary_test` or model swapping logic within `agent/`.
- Recommended next step: Move the canary testing and fail-fast fallback logic into the `DeepResearcher` class itself, making it configurable.
- Blocks future work? no

- Blindspot ID: BS-UNK-005
- Title: Do analysis patches overfit?
- Category: Known Unknowns
- Severity: high
- Confidence: low
- Status: unresolved unknown
- Why this is a blindspot: `analysis/analyze_results.py` generates auto-patches (`prompt_patch.txt`) based entirely on self-reported `solved` booleans and trace metadata.
- Evidence: `analysis/analyze_results.py:analyze`: `solved = sum(1 for t in traces if t.get("solved", False))`
- What could be misleading: A user reading the analysis output (e.g., "50% success rate") assumes the generated system prompt patches are optimizing for true mathematical accuracy against a ground truth dataset like AIMO.
- Potential consequence: Because the agent conflates "code execution success" with "mathematical correctness" (BS-VER-001) and forces consensus string matching without normalization (BS-ANS-001), the auto-patcher is highly vulnerable to optimizing for the wrong thing.
- How to verify or falsify it: Cross-reference the `research_data.jsonl` output with the actual AIMO evaluation dataset and calculate the discrepancy between the self-reported `solved` boolean and the actual success rate.
- Recommended next step: Integrate an independent ground-truth evaluation step into `analyze_results.py` before generating `prompt_patch.txt`.
- Blocks future work? yes
