# Verification Blindspots

This document exposes execution vs. truth conflation, overloaded booleans, natural language critique biases, and unverifiable parts of the pipeline.

- Blindspot ID: BS-VER-001
- Title: Execution Conflated with Correctness
- Category: Verification
- Severity: high
- Confidence: high
- Status: fact
- Why this is a blindspot: The architecture explicitly treats a script executing without throwing a Python exception as equivalent to the mathematical correctness of an intermediate step or final answer.
- Evidence:
  - `agent/sandbox.py:run_verification` returns a boolean (`all_passed`) which is `True` if `returncode == 0` for all blocks.
  - `agent/deep_researcher.py:_majority_vote`: `weight = 2 if passed else 1`
  - The `Attempt` dataclass tracks this single boolean as `verification_passed: bool`.
- What could be misleading: A reader of the logs or metrics (`verification_passed: True`) might assume the agent wrote a programmatic check (like `assert`) or that an external verifier validated the logic. In reality, ````python print("Hello") ```` will be marked `verification_passed: True`.
- Potential consequence: The agent frequently rewards hallucinated or trivial code blocks simply because they don't crash the interpreter, granting bad reasoning traces undue voting weight.
- How to verify or falsify it: Inject `print("This proves nothing")` as the only code block and observe that `run_verification` returns `True`.
- Recommended next step: Refactor the `Attempt` dataclass to separate `code_executed_successfully` from `mathematically_verified`. Use a strongly typed enum instead of overloaded booleans.
- Blocks future work? yes

- Blindspot ID: BS-VER-002
- Title: Confirmation Bias in NL Verifier
- Category: Verification
- Severity: high
- Confidence: high
- Status: fact
- Why this is a blindspot: After a consensus answer is found via majority vote, the agent optionally prompts the same LLM to review the winning reasoning trace for "logical errors."
- Evidence:
  - `agent/prompts.py:NL_VERIFY_PROMPT` and `extract_nl_verdict`
  - `agent/deep_researcher.py:_nl_verify` and `_research_problem_inner`
- What could be misleading: The system explicitly states it is performing a "Natural Language Review" to verify correctness, implying a rigorous critique. However, the same underlying model used to generate flawed reasoning is immediately asked to critique it, often resulting in rubber-stamp approvals.
- Potential consequence: The agent repeatedly self-validates incorrect logic, leading to confirmation bias, while "successful NL Verification" metrics remain high, masking the issue. Furthermore, if the NL verifier model hallucinates the output format and fails to include the exact `VERDICT:` strings, `extract_nl_verdict` silently defaults to `True`.
- How to verify or falsify it: Feed a known-bad solution generated by the model into `_nl_verify` and observe how frequently it still returns `VERDICT: CORRECT` or defaults to `True`.
- Recommended next step: Route NL Verification to a more capable reasoning model (e.g., DeepSeek-R1, O1) or implement programmatic checks. Address the pass-through failure in `extract_nl_verdict`.
- Blocks future work? no

- Blindspot ID: BS-VER-003
- Title: Coarse Trace Schema `verification_passed`
- Category: Verification
- Severity: medium
- Confidence: high
- Status: fact
- Why this is a blindspot: The `Attempt` schema uses a single boolean to track verification.
- Evidence:
  - `agent/deep_researcher.py:Attempt` -> `verification_passed: bool`
- What could be misleading: By collapsing the state space into a single boolean, the analysis pipeline (`analyze_results.py`) cannot accurately distinguish between "failed because of a timeout," "failed because of a syntax error," and "failed because the math assertion failed."
- Potential consequence: Data-driven auto-patching and DPO training rely on noisy signals, mischaracterizing the failure modes of the agent.
- How to verify or falsify it: Observe that `analyze_results.py` has to hackishly parse the `verification_output` string to count specific failure patterns (like `SyntaxError` vs `TIMEOUT`) because the schema itself is too coarse.
- Recommended next step: Implement the enum states proposed in `docs/VERIFICATION_AUDIT.md`.
- Blocks future work? no
